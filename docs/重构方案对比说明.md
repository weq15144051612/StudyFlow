# StudyFlow 项目任务排序策略重构方案对比说明

## 1. 文档概述

### 1.1 目的
本文档旨在对 StudyFlow 项目中 `TaskServiceImpl` 的优先级排序硬编码问题，提供两种重构方案的详细对比分析，为架构决策提供依据。

### 1.2 背景
当前 `TaskServiceImpl.listTasksByPriority()` 方法中的排序规则硬编码，导致：
- 扩展新排序规则需要修改核心业务代码
- 违反开闭原则
- 难以支持动态切换排序策略

### 1.3 评审范围
| 对比维度 | 说明 |
|---------|------|
| 影响范围 | 改动文件数量及影响程度 |
| 抽象层设计 | 是否需要新增接口/抽象类 |
| 测试适配 | 已有测试的迁移难度 |
| 项目适配 | 是否符合当前项目规模 |
| 验收要求 | 是否满足实验三设计模式要求 |
| 扩展能力 | 对未来需求的支持程度 |

## 2. 方案 A：低改动快速方案

### 2.1 核心思路
将排序规则提取为可配置的成员变量，通过构造函数注入，**不引入新设计模式**。

### 2.2 架构设计

**类图（核心改动）：**

```
┌─────────────────────────────────┐
│      TaskServiceImpl            │
├─────────────────────────────────┤
│ - taskRepository: Repository    │
│ - priorityOrder: Record<>       │  ← 新增可配置属性
├─────────────────────────────────┤
│ + constructor(repo, order?)     │  ← 构造函数支持可选参数
│ + listTasksByPriority(): Task[] │  ← 使用成员变量排序
└─────────────────────────────────┘
```

### 2.3 关键代码

```typescript
// src/infrastructure/TaskServiceImpl.ts
export class TaskServiceImpl implements TaskService {
  private priorityOrder: Record<Priority, number>;

  constructor(
    private readonly taskRepository: TaskRepository,
    priorityOrder: Record<Priority, number> = { 
      [Priority.HIGH]: 0, 
      [Priority.MEDIUM]: 1, 
      [Priority.LOW]: 2 
    }
  ) {
    this.priorityOrder = priorityOrder;
  }

  listTasksByPriority(userId: string): Task[] {
    const tasks = this.taskRepository.findByUserId(userId);
    return tasks.sort((a, b) => 
      this.priorityOrder[a.priority] - this.priorityOrder[b.priority]
    );
  }
}
```

### 2.4 使用示例

```typescript
// 默认排序（高→中→低）
const service = new TaskServiceImpl(repository);

// 自定义排序（低→中→高）
const service = new TaskServiceImpl(repository, {
  [Priority.HIGH]: 2,
  [Priority.MEDIUM]: 1,
  [Priority.LOW]: 0
});
```

## 3. 方案 B：策略模式方案

### 3.1 核心思路
引入**策略模式**，定义 `TaskSortStrategy` 接口和具体策略实现，实现排序算法与业务逻辑的完全解耦。

### 3.2 架构设计

**类图：**

```
┌─────────────────────────────────┐
│    TaskSortStrategy (接口)      │
├─────────────────────────────────┤
│ + sort(tasks: Task[]): Task[]   │
└─────────────────────────────────┘
        ▲
        │ 实现
┌───────┴─────────┬─────────────────┐
│ PrioritySort    │ DueDateSort     │  ← 可扩展更多策略
│ Strategy        │ Strategy        │
└─────────────────┴─────────────────┘
        ▲
        │ 依赖注入
┌─────────────────────────────────┐
│      TaskServiceImpl            │
├─────────────────────────────────┤
│ - taskRepository: Repository    │
│ - sortStrategy: Strategy        │
├─────────────────────────────────┤
│ + constructor(repo, strategy?)  │
│ + listTasksByPriority(): Task[] │
│ + setSortStrategy(): void       │
└─────────────────────────────────┘
```

### 3.3 关键代码

**策略接口：**
```typescript
// src/domain/TaskSortStrategy.ts
export interface TaskSortStrategy {
  sort(tasks: Task[]): Task[];
}
```

**具体策略实现：**
```typescript
// src/domain/strategy/PrioritySortStrategy.ts
export class PrioritySortStrategy implements TaskSortStrategy {
  private priorityOrder: Record<Priority, number>;

  constructor(order?: Record<Priority, number>) {
    this.priorityOrder = order || { 
      [Priority.HIGH]: 0, 
      [Priority.MEDIUM]: 1, 
      [Priority.LOW]: 2 
    };
  }

  sort(tasks: Task[]): Task[] {
    return [...tasks].sort((a, b) => 
      this.priorityOrder[a.priority] - this.priorityOrder[b.priority]
    );
  }
}
```

**修改后的 Service：**
```typescript
// src/infrastructure/TaskServiceImpl.ts
export class TaskServiceImpl implements TaskService {
  constructor(
    private readonly taskRepository: TaskRepository,
    private readonly sortStrategy: TaskSortStrategy = new PrioritySortStrategy()
  ) {}

  listTasksByPriority(userId: string): Task[] {
    const tasks = this.taskRepository.findByUserId(userId);
    return this.sortStrategy.sort(tasks);
  }
}
```

### 3.4 使用示例

```typescript
// 使用优先级排序（默认）
const service1 = new TaskServiceImpl(repository);

// 使用截止日期排序
const service2 = new TaskServiceImpl(repository, new DueDateSortStrategy());

// 运行时切换策略
service.setSortStrategy(new CustomSortStrategy());
```

## 4. 方案对比分析

### 4.1 对比矩阵

| 维度 | 方案 A | 方案 B |
|-----|--------|--------|
| **改动文件数** | 1 个 | 3-4 个 |
| **新增抽象层** | 否 | 是（接口+策略类） |
| **测试迁移难度** | 低 | 中 |
| **项目规模适配** | 适合小型项目 | 适合中大型项目 |
| **设计模式引入** | 否 | 是（策略模式） |
| **实验验收满足** | 不满足 | 满足 |
| **提醒策略扩展支持** | 弱 | 强 |
| **开闭原则符合** | 部分符合 | 完全符合 |
| **代码复用性** | 低 | 高 |

### 4.2 详细对比说明

#### 4.2.1 影响范围对比

**方案 A**：
- 仅修改 `TaskServiceImpl.ts`
- 改动集中在构造函数和排序方法
- 其他方法完全不变
- **影响范围极小**

**方案 B**：
- 新增文件：`TaskSortStrategy.ts`、`PrioritySortStrategy.ts`
- 修改文件：`TaskServiceImpl.ts`
- 需要更新 `domain/index.ts` 导出接口
- **影响范围较大但职责清晰**

#### 4.2.2 抽象层设计对比

**方案 A**：
- 无需新增抽象层
- 保持原有类结构
- 适合简单场景

**方案 B**：
- 新增 `TaskSortStrategy` 接口
- 定义策略契约
- 为未来扩展提供清晰边界

#### 4.2.3 测试适配对比

**方案 A**：
- 现有测试基本无需修改
- 只需新增一个测试用例验证自定义排序
- **迁移成本低**

**方案 B**：
- 需要为策略类编写独立单元测试
- Service 测试需注入 mock 策略
- 测试数量增加但更聚焦
- **迁移成本中等**

#### 4.2.4 扩展能力对比

**方案 A**：
- 仅支持优先级排序规则的配置
- 新增排序算法需要修改 Service
- **扩展能力有限**

**方案 B**：
- 支持任意排序算法扩展
- 新增策略只需实现接口
- 可复用框架扩展提醒策略等
- **扩展能力强**

## 5. 推荐方案及理由

### 5.1 推荐选择

**推荐方案：方案 B（策略模式）**

### 5.2 推荐理由

| 理由 | 说明 |
|-----|------|
| **实验要求满足** | 必须引入设计模式，方案 B 符合验收标准 |
| **开闭原则** | 对扩展开放（新增策略），对修改关闭（不改原有代码） |
| **扩展性保障** | 为提醒策略等未来需求预留扩展点 |
| **代码质量** | 职责分离、易于测试、可读性高 |
| **学习价值** | 实践策略模式，加深对设计模式的理解 |

### 5.3 实施建议

1. **第一步**：创建 `TaskSortStrategy` 接口
2. **第二步**：实现 `PrioritySortStrategy`（迁移现有排序逻辑）
3. **第三步**：修改 `TaskServiceImpl` 注入策略接口
4. **第四步**：为策略类编写单元测试
5. **第五步**：更新 `domain/index.ts` 导出接口
