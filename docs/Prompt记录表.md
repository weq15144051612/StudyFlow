# Prompt 与人工修订记录表

## 记录 1: 需求整理

### Prompt (发送给AI)
```
请将以下课程实验项目描述整理为软件需求清单。
输出要求包括：
- 用户角色
- 用户故事
- 功能需求
- 非功能需求
- 可测试的验收标准

项目描述：StudyFlow 是一个面向学生的学习任务管理系统，支持个人任务管理与学习进度跟踪。
```

### AI 输出
```
用户角色：
- 个人用户（学生）

用户故事：
1. 作为一个学生，我想要创建学习任务，这样我可以规划我的学习计划
2. 作为一个学生，我想要设置任务优先级，这样我可以确定任务的重要程度
3. 作为一个学生，我想要设置提醒，这样我不会错过重要的截止日期
4. 作为一个学生，我想要更新任务状态，这样我可以跟踪我的进度
5. 作为一个学生，我想要查看统计，这样我可以了解我的学习情况

功能需求：
1. 任务创建功能
2. 任务查询功能
3. 任务状态管理
4. 提醒功能
5. 任务统计

非功能需求：
1. 性能要求
2. 可用性要求
3. 安全性要求

验收标准：
1. 系统可以成功创建任务
2. 系统可以查询任务
3. 系统可以更新状态
...
```

### 人工修订
- **修订内容**：
  - 功能需求数量从5条扩展到8条，增加了任务列表管理、进度追踪等功能
  - 非功能需求增加为5条，包含可维护性和可扩展性
  - 验收标准从3条增加到5条，增加了数据一致性验证

- **修订理由**：
  - 原始需求过于简单，需要更完整的功能覆盖
  - 添加了更具体的业务规则，如截止日期不能早于创建时间
  - 补充了不变量约束

## 记录 2: 领域建模

### Prompt (发送给AI)
```
请基于以下实体描述，帮我进行领域建模：
- User（用户）
- Task（任务）
- TaskList（任务列表）
- ReminderPolicy（提醒策略）
- ProgressReport（进度报告）

请分析：
1. 哪些应该是实体，哪些应该是值对象
2. 实体之间应该是什么关系
3. 给出至少3条关键不变量
```

### AI 输出
```
实体分析：
- User: 实体（有唯一ID，可独立存在）
- Task: 实体（有唯一ID，可独立存在）
- TaskList: 实体（有唯一ID，可独立存在）
- ReminderPolicy: 值对象（不可变，没有唯一标识）
- ProgressReport: 值对象（统计信息，不可变）

关系：
- User 1:N Task
- User 1:N TaskList
- TaskList 1:N Task

不变量：
1. 任务截止日期 >= 创建日期
2. 已完成任务状态不可逆
3. 高优先级任务必须设置截止日期
```

### 人工修订
- **修订内容**：
  - 将ProgressReport改为值对象（原来是实体）
  - 添加了实体关系的具体说明（User与Task是一对多关系）
  - 补充了不变量3条（原来AI只给了3条，但描述不够精确）

- **修订理由**：
  - ProgressReport是根据任务数据生成的统计报告，本身不可变，符合值对象特征
  - 需要明确实体间关系的基数（ Cardinality）
  - 不变量描述需要更精确，包含业务含义

## 记录 3: 项目结构设计

### Prompt (发送给AI)
```
我需要为一个学习任务管理系统设计项目结构。系统需要包含：
- 领域层（domain）
- 应用层（application）
- 基础设施层（infrastructure）
- 接口层（interfaces）

请给出具体的目录结构和每个模块的职责说明。
```

### AI 输出
```
src/
├── domain/          # 领域层
│   ├── entities/    # 实体
│   ├── valueObjects/# 值对象
│   └── services/    # 领域服务
├── application/     # 应用层
│   ├── useCases/    # 用例
│   └── dto/         # 数据传输对象
├── infrastructure/  # 基础设施层
│   ├── persistence/ # 持久化
│   └── external/    # 外部服务
└── interfaces/      # 接口层
    ├── api/         # API控制器
    └── cli/         # 命令行接口
```

### 人工修订
- **修订内容**：
  - 将domain下的entities/和valueObjects/合并，直接放实体文件（简化结构）
  - 在domain/中添加了index.ts统一导出
  - 将services从domain独立到TaskService.ts文件
  - 在application/中添加了DTO.ts文件
  - 删除了cli/目录，因为本项目不需要命令行接口

- **修订理由**：
  - 实体和值对象数量不多，不需要单独目录
  - 领域服务接口和实现分开更清晰
  - 简化目录结构，提高可维护性
  - 根据实际需求调整，不是照搬AI建议

## 记录 4: 接口设计

### Prompt (发送给AI)
```
请帮我设计TaskService的接口，包含以下功能：
- 创建任务
- 完成任务
- 按优先级查询任务
- 查询到期任务

使用TypeScript接口定义。
```

### AI 输出
```typescript
interface TaskService {
  createTask(title: string, priority: string, dueDate: Date): Task;
  completeTask(taskId: string): Task;
  getTasksByPriority(userId: string): Task[];
  getTasksByDueDate(date: Date): Task[];
}
```

### 人工修订
- **修订内容**：
  - createTask使用Command模式，传入CreateTaskCommand对象
  - 添加了updateTaskStatus方法
  - getTasksByPriority改为listTasksByPriority
  - 添加了getTaskById方法用于查询单个任务
  - 添加了TaskRepository接口定义

- **修订理由**：
  - 使用Command模式更清晰，便于扩展参数
  - 需要支持部分更新任务状态（不仅仅是完成）
  - 查询单个任务是常见需求
  - 分离接口和实现，符合依赖倒置原则

## 总结

### AI 的价值
1. 快速生成基础结构和模板
2. 提供常见的最佳实践建议
3. 加速需求文档的初稿撰写

### 人工判断的重要性
1. 需求需要根据实际情况调整，不能直接使用AI输出
2. 架构设计需要权衡复杂度与实际需求
3. 接口设计需要考虑可扩展性和易用性
4. 需要识别哪些建议适合当前项目规模

### AI 与人工的协作方式
1. **先独立思考**：人工先形成初步方案
2. **AI补充完善**：AI提供建议和参考
3. **人工决策**：最终由人工决定采纳哪些建议
4. **迭代优化**：在实践中不断调整
