# 实验一报告：AI 辅助需求建模与系统骨架构造

## 1. 实验目标

1. 学会把自然语言需求转化为可实现的功能列表与约束条件
2. 学会借助 AI 提炼用户故事、领域对象和模块划分
3. 完成项目初始化与基本目录结构设计
4. 建立第一版需求基线与架构基线

## 2. 实验环境

1. **硬件环境**：普通 PC 计算机
2. **软件环境**：Windows/macOS/Linux、VS Code、Node.js 20+、Git
3. **工具依赖**：Vitest（单元测试）、ESLint、Prettier
4. **AI 辅助工具**：GitHub Copilot / ChatGPT / VS Code Chat
5. **项目基础**：实验二完成的 StudyFlow 学习任务管理系统代码

## 6. 核心代码或设计说明

### 5.1 Task 实体核心实现

```typescript
export class Task {
  public readonly id: string;
  public status: TaskStatus;
  public completedAt: Date | null;

  constructor(props: TaskProps) {
    this.id = props.id || uuidv4();
    this.status = props.status || TaskStatus.TODO;
    this.completedAt = null;
  }

  public complete(): void {
    if (this.status === TaskStatus.COMPLETED) {
      throw new Error('Task is already completed');
    }
    this.status = TaskStatus.COMPLETED;
    this.completedAt = new Date();
  }

  public updateStatus(newStatus: TaskStatus): void {
    if (this.status === TaskStatus.COMPLETED && newStatus === TaskStatus.TODO) {
      throw new Error('Completed task cannot be changed back to TODO');
    }
    this.status = newStatus;
  }
}
```

**设计理由**:
- 使用不可变的 `id` 确保任务唯一性
- 在实体内部封装状态转换逻辑，保证业务规则（不变量）的执行
- 通过方法而非直接属性访问来修改状态，实现封装

### 5.2 分层架构设计

```
┌─────────────────────────────────────┐
│        interfaces/                  │  ← 接口层：处理外部请求
├─────────────────────────────────────┤
│        application/                 │  ← 应用层：用例编排
├─────────────────────────────────────┤
│        domain/                      │  ← 领域层：核心业务逻辑
├─────────────────────────────────────┤
│        infrastructure/               │  ← 基础设施层：持久化
└─────────────────────────────────────┘
```

**设计理由**:
- **领域层** 保持纯净，不依赖任何外部框架
- **应用层** 编排领域对象，实现用例
- **基础设施层** 实现接口依赖，便于替换
- **接口层** 处理HTTP、CLI等外部交互

## 7. 遇到的问题与解决过程

### 问题 1: 需求过于宽泛

**描述**: 初始需求描述"学习任务管理系统"过于抽象，难以直接转化为代码。

**解决**: 使用AI辅助将自然语言需求分解为具体的用户故事、功能点和验收标准，然后人工审核和补充。

### 问题 2: 实体与值对象的区分

**描述**: 最初将ProgressReport设计为实体，但经过分析发现它只是统计数据，应该不可变。

**解决**: 查阅DDD相关资料，将ProgressReport改为值对象，确保其不可变特性。

### 问题 3: 项目结构选择

**描述**: AI建议的项目结构较为复杂，不适合小型项目。

**解决**: 根据实际项目规模和需求，简化了目录结构，删除了不必要的分层。

## 8. 个人反思

### AI 在独立开发中的价值

1. **加速需求文档初稿撰写**: AI能够快速生成需求框架，减少从零开始的工作量
2. **提供最佳实践建议**: AI可以推荐常见的架构模式和设计方法
3. **辅助代码生成**: 对于重复性的代码，AI可以提高生成效率

### AI 的局限

1. **缺乏对具体业务的理解**: AI难以准确把握业务优先级和关键约束
2. **过度设计倾向**: AI倾向于提供"大而全"的解决方案
3. **无法替代人工判断**: 最终的设计决策需要人工做出，特别是涉及权衡时

### 关键经验

1. **先独立思考，再使用AI**: 不应该依赖AI来形成初始方案
2. **AI是工具，不是答案**: AI的建议需要人工审核和调整
3. **保持工程纪律**: 即使使用AI，也要遵循软件构造的基本原则
