# StudyFlow 领域模型说明

## 1. 核心实体

### 1.1 User (用户)
```
User {
  id: string (唯一标识)
  name: string
  email: string
  createdAt: Date
  updatedAt: Date
}
```

### 1.2 Task (任务) - 核心实体
```
Task {
  id: string (唯一标识)
  userId: string (所属用户ID)
  title: string (任务标题)
  description: string (任务描述)
  priority: Priority (优先级)
  status: TaskStatus (任务状态)
  dueDate: Date (截止日期)
  createdAt: Date
  updatedAt: Date
  completedAt: Date | null
}
```

### 1.3 TaskList (任务列表)
```
TaskList {
  id: string (唯一标识)
  userId: string (所属用户ID)
  name: string (列表名称)
  description: string (列表描述)
  taskIds: string[] (包含的任务ID列表)
  createdAt: Date
  updatedAt: Date
}
```

## 2. 值对象 (Value Objects)

### 2.1 Priority (优先级)
```
Priority: enum {
  HIGH = "high",     // 高优先级
  MEDIUM = "medium", // 中优先级
  LOW = "low"        // 低优先级
}
```

### 2.2 TaskStatus (任务状态)
```
TaskStatus: enum {
  TODO = "todo",       // 待办
  IN_PROGRESS = "in_progress", // 进行中
  COMPLETED = "completed"      // 已完成
}
```

### 2.3 ReminderPolicy (提醒策略)
```
ReminderPolicy {
  type: ReminderType
  daysBefore: number  // 提前天数
  isInstant: boolean  // 是否即时提醒
}
```

### 2.4 ProgressReport (进度报告)
```
ProgressReport {
  userId: string
  totalTasks: number
  completedTasks: number
  completionRate: number
  generatedAt: Date
}
```

## 4. 关键不变量 (Business Invariants)

### Invariant 1: 截止日期约束
```
∀ task ∈ Task:
  task.dueDate >= task.createdAt
```
**说明**：任务截止时间不能早于创建时间

### Invariant 2: 状态不可逆
```
∀ task ∈ Task WHERE task.status = COMPLETED:
  task.status cannot be changed to TODO
```
**说明**：已完成的任务不能再次被修改为"待办"状态

### Invariant 3: 高优先级必须有时限
```
∀ task ∈ Task WHERE task.priority = HIGH:
  task.dueDate must be set (not null)
```
**说明**：高优先级任务必须具有截止日期

## 5. 领域服务

### TaskService
```typescript
interface TaskService {
  // 创建任务
  createTask(input: CreateTaskCommand): Task;

  // 完成任务
  completeTask(taskId: string): Task;

  // 更新任务状态
  updateTaskStatus(taskId: string, newStatus: TaskStatus): Task;

  // 按优先级查询用户任务
  listTasksByPriority(userId: string): Task[];

  // 查询到期任务
  listDueTasks(date: Date): Task[];
}
```

## 6. 模块职责说明

### domain/ (领域层)
- **职责**：封装核心业务逻辑和规则
- **包含**：实体、值对象、领域服务接口、领域事件

### application/ (应用层)
- **职责**：编排领域对象完成用例，协调任务
- **包含**：用例类、DTO、事务脚本

### infrastructure/ (基础设施层)
- **职责**：提供持久化、技术服务和外部适配器
- **包含**：数据库实现、消息队列、第三方服务

### interfaces/ (接口层)
- **职责**：处理外部请求和展示结果
- **包含**：REST控制器、CLI处理器、DTO转换器

## 7. 建模决策说明

### 为什么Task是实体而Priority是值对象？
- **Task** 具有独立生命周期和唯一标识，可独立存在和追踪
- **Priority** 只是枚举类型，不可变，没有独立标识

### 为什么ReminderPolicy是值对象？
- 提醒策略是不可变的，一旦创建不可修改
- 策略本身没有唯一标识，用于表达规则

### 为什么User与Task是一对多关系？
- 一个用户可以拥有多个任务
- 每个任务只属于一个用户
- 任务不能独立于用户存在（必须有userId）
